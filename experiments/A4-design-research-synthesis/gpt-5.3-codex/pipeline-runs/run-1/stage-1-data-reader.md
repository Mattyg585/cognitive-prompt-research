## Participant Observations

### P1 — Sarah (Engineering Manager, 85-person company)
**Context:** Team adopted enthusiastically; engineering-heavy workflow with coordination in Slack.

**Key quotes:**
- “Updating a task takes four clicks... In Slack I just say ‘done with the auth thing’ and everyone knows.”
- “If people aren’t updating tasks, the report is fiction.”
- “We use maybe 30% of the features... the rest is for a different kind of company.”

**Behavioral observations:**
- Team shifted from TaskFlow updates back to Slack thread coordination.
- Sarah stopped using weekly reports after confidence dropped.

**Pain points:**
- Update interaction friction.
- Report unreliability due to stale task data.
- Perceived over-featured product for team context.

**Workarounds:**
- Slack status communication replacing structured task updates.

**Emotional moments:**
- Pragmatic resignation (“I know that’s terrible for tracking but it’s honest”).

**Contradictions / tensions:**
- Values tracking/reporting but uses informal communication for speed.

**Noteworthy details:**
- Strong feature praise coexists with sustained behavior drift away from product.

### P2 — Marcus (Product Designer, 120-person company)
**Context:** High individual engagement; low team-level engagement.

**Key quotes:**
- “The ‘team view’ is really just Marcus’s view.”
- “I want who is working on what right now... default view shows everything.”
- “I gave up after twenty minutes... I’m a designer, not a systems architect.”

**Behavioral observations:**
- Maintains personal task hygiene while teammates do not.
- Built custom views, but default path remains noisy.

**Pain points:**
- Low default signal-to-noise.
- Setup burden for useful views.
- Automation complexity mismatch with role.

**Workarounds:**
- Manual custom filtering and saved views.

**Emotional moments:**
- Isolation frustration (solo adherence in team tool).

**Contradictions / tensions:**
- Uses tool “religiously” yet still reports low team utility.

**Noteworthy details:**
- Engagement by one user can mask poor team-level utility.

### P3 — Alex (VP Operations, 200-person company)
**Context:** Buyer/leader perspective focused on cross-team visibility.

**Key quotes:**
- “We wanted one place to see everything... we’re at maybe 60% compliance.”
- “TaskFlow is an extra tab they have to remember to update.”
- “TaskFlow wants to be the center. It should be the lens.”

**Behavioral observations:**
- Organization attempted broad adoption but plateaued below full compliance.
- Integration experiments did not resolve workflow mismatch.

**Pain points:**
- Incomplete participation undermines visibility objective.
- Existing integration behavior seen as noisy (“firehose”).

**Workarounds:**
- Continued reliance on function-specific tools plus partial TaskFlow use.

**Emotional moments:**
- Strategic frustration with adoption ceiling.

**Contradictions / tensions:**
- Wants centralized visibility, but user behavior remains decentralized.

**Noteworthy details:**
- Explicit “lens vs center” framing from leadership perspective.

### P4 — Deepa (Project Manager, 90-person company)
**Context:** Team champion and operational enforcer.

**Key quotes:**
- “I spend about 30 minutes a day nagging people to update tasks.”
- “I’m the human integration layer.”
- “Everyone turned [notifications] off in the first week.”

**Behavioral observations:**
- Performs manual status updates during standup on behalf of team.
- Compensates for low teammate engagement by direct enforcement.

**Pain points:**
- Role displacement from project management to compliance management.
- Notification system creates all-or-nothing behavior.

**Workarounds:**
- Standup-based manual synchronization.

**Emotional moments:**
- Strong irony and fatigue language about overhead burden.

**Contradictions / tensions:**
- Product is “objectively better” than spreadsheet but lower actual participation.

**Noteworthy details:**
- Manual workaround maintains data but defeats self-service intent.

### P5 — Raj (Senior Engineer, 150-person company)
**Context:** Individual contributor with personal productivity system.

**Key quotes:**
- “I went back to a text file.”
- “47 tasks across 6 projects... couldn’t tell which ones actually mattered.”
- “I’m using TaskFlow as an inbox, not a workspace.”

**Behavioral observations:**
- Uses TaskFlow mainly to check assigned work, then re-tracks elsewhere.
- Continues light mobile usage despite desktop disengagement.

**Pain points:**
- Priority ambiguity in overloaded task lists.
- Completion flow friction even in mobile context.

**Workarounds:**
- Personal todo.txt as true workspace.

**Emotional moments:**
- Clear rejection of system complexity with selective praise of mobile app.

**Contradictions / tensions:**
- Maintains interaction with product while relocating actual planning externally.

**Noteworthy details:**
- “Inbox not workspace” describes limited utility role rather than full abandonment.

### P6 — Nina (Marketing Lead, 70-person company)
**Context:** Small team lead; appreciates planning features but questions fit.

**Key quotes:**
- “Timeline view... best I’ve seen.”
- “TaskFlow gives us a Ferrari when we need a bicycle.”
- “The gap between the admin experience and the user experience is where adoption dies.”

**Behavioral observations:**
- High setup engagement from lead; low day-to-day engagement from team members.

**Pain points:**
- Overwhelming onboarding and configuration load.
- Product complexity exceeds team operational needs.

**Workarounds:**
- None explicit beyond reduced depth of ongoing use.

**Emotional moments:**
- Simultaneous enthusiasm for specific features and concern about practical fit.

**Contradictions / tensions:**
- Product excellence in planning features coexists with poor team usability.

**Noteworthy details:**
- Names admin-vs-contributor split as central adoption mechanism.
